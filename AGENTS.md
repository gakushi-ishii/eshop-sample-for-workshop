# Git / GitHub エージェント運用ルール

このファイルは、AI エージェントによる Git / GitHub 操作のルールを定める。
実行環境・依存関係・品質ゲートは `.github/copilot-instructions.md` を参照する。

## ブランチ運用

GitHub Copilot App が作成するワークツリーの自動生成ブランチは、作業開始時に
チームの命名規約へ合わせる。

### 適用条件

次の条件をすべて満たす場合に限り、最初の追跡対象ファイルを変更する直前に一度だけ
ブランチ名を確認する。

- ユーザーの依頼がファイル変更を伴う。
- GitHub Copilot App の自動生成ワークツリーブランチである。
- `ask_user` と `rename_branch` が利用できる。
- ブランチがまだリモートへ公開されておらず、関連する PR も存在しない。

相談、調査、レビュー、読み取り、コマンド実行だけの依頼では確認しない。

### 自動リネームしないケース

- 現在のブランチが `main` などの既定・保護ブランチである。
- ワークツリーではなく、既存ブランチを開いた in-place セッションである。
- すでに `feature/<issue番号>-<説明>` または
  `bugfix/<issue番号>-<説明>` の形式である。
- upstream、リモートブランチ、または関連する PR が存在する。
- 必要な対話ツールまたはリネームツールが利用できない。
- ユーザーが「リネームしない」を選択した。

自動リネームを行わないことは、保護ブランチへのコミットや push を許可するものではない。
ツールが利用できない場合も、シェルコマンドによるリネームで代替しない。
既定・保護ブランチ上で変更を伴う依頼を受けた場合は、ユーザーがそのブランチで
作業する意図を明示していない限り、編集前に別ワークツリーまたは作業ブランチの利用を確認する。

### 手順

1. `ask_user` で次の選択肢を提示する。
   - `リネームしない`
   - `新機能（feature）`
   - `バグ修正（bugfix）`
2. リネームする場合は、`ask_user` に `000（Issueなし）` を選択肢として提示し、
   自由入力で issue 番号も受け付ける。
   - `#123` は `123` に正規化する。
   - 数字以外を含む値は使用せず、再入力を求める。
   - issue がない場合は `000` を使用する。
3. ユーザーの依頼内容から、英小文字の kebab-case で 3〜5 語程度の説明を作る。
   記号や日本語は含めない。
4. 次の形式でブランチ名をドラフトする。

   ```text
   <prefix>/<issue番号>-<簡潔な説明>
   ```

   - 新機能: `feature`（例: `feature/1234-add-cart-api`）
   - バグ修正: `bugfix`（例: `bugfix/000-prevent-search-crash`）
5. `ask_user` で、ドラフトを示して次の選択肢を提示する。
   - `このブランチ名でリネーム`
   - `説明を修正`
   - `リネームしない`
6. 承認された場合だけ `rename_branch` を一度実行する。説明が修正された場合は、
   新しいドラフトを再提示して承認を得る。

## Issue 対応

- バグ修正では、Issue の再現手順と期待結果を確認し、まず問題を再現するテストを追加する。
- PR 本文では、受け入れ条件ごとに対応する変更とテストを対応付ける。

## コミット

- `git commit` の前に、日本語のコミットメッセージをドラフトしてユーザーへ提示する。
- 件名は 50 文字程度の要約とし、必要なら本文に何を・なぜ変更したかを書く。
- `ask_user` で `このメッセージでコミット`、`メッセージを修正`、
  `コミットしない` を提示する。
- ユーザーが承認するまでコミットしない。
- 「リネームしない」の選択を、コミットの承認として扱わない。

## Push / Pull Request

- ユーザーの明示的な依頼なしに push や Pull Request 作成を行わない。
- upstream または Pull Request が存在するブランチを自動でリネームしない。
- force push、rebase、履歴の書き換え、破壊的な Git 操作は、ユーザーの明示的な承認なしに行わない。
