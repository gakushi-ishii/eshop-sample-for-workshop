# Lab 04: 新しいセッションへ引き継ぐ

**テーマ:** 会話履歴ではなく、リポジトリに残した情報でコンテキストを引き継ぐ

## シナリオ

Cloud Agent が PR を作る間に、別の担当者が作業を引き継ぐ状況を再現する。
過去の会話を見られない新しいセッションが、リポジトリの情報だけで
「実行環境・技術スタック・依存ルール・実装の方針」を復元できるかを確認する。

これが「AI Ready」の効果。ガードレールと判断記録があるから、会話が消えても文脈が残る。

## 前提条件

- Lab 03 で Cloud Agent に委託済みで、その完了を待っている状態。
- `main` に Lab 02 の変更がマージされていること。

## 手順

### 1. 新しいセッションを作成する

`main` から **New worktree** で新しいセッションを作成する（Lab 01 の会話は引き継がない）。

### 2. リポジトリだけで状況を説明させる

```text
このリポジトリの前提を、過去の会話を使わずに説明してください。
1. 実行環境と技術スタック（バージョンを含む）。
2. 依存パッケージを追加するときに守るルールと、その根拠ファイル。
3. ファイル変更を伴うローカル作業で、ルールと Git 手順をどのファイルが担うか。
4. 最近追加された商品詳細ルーティングで、なぜ react-router-dom を採用したか。
それぞれ、どのファイルを根拠にしたかを示してください。
```

**期待する結果:**
- Node 20 / React 18 / Vite 6 / TypeScript などを `.github/copilot-instructions.md` と `.node-version`、`package.json` から説明する。
- 依存ルールを `copilot-instructions.md` と `add-dependency-safely` スキルから説明する。
- SDLC ルールを `AGENTS.md`、具体的な Git 手順を `git-workflow` スキルから説明する。
- ルーティング採用理由を `docs/decisions/` の ADR から説明する。

## 期待する結果 / 残る成果物

- 新しいセッションが、会話履歴なしで前提と判断を再構成できたことの確認。

> もし説明が曖昧なら、それは「まだ言語化されていない暗黙知」が残っているサイン。
> 何を追記すべきかを考えることが、AI Ready 化の継続的な改善につながる。
