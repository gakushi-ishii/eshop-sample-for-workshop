# 設計判断の記録（ADR）

このディレクトリには、後から「なぜこうした？変えていい？」と問われる設計判断を短く記録する。

## 目的

- 判断の理由と却下した代替案を残し、会話が消えても文脈を再利用できるようにする。
- コーディングエージェントが読み込む前提として、1 件 30 行以内に収める。

## いつ書くか

- 依存パッケージの追加・更新（例: ルーティングライブラリの採用）
- 技術方式の選択（例: 画面遷移の実現方法）
- 「6 ヶ月後に別の人が理由を問う」判断

一時的な値（SKU、微調整など）は ADR にしない。

## フォーマット

```markdown
# ADR-NNNN: <タイトル>

- ステータス: 採用 / 却下 / 置換
- 日付: YYYY-MM-DD

## 背景
<何を決める必要があったか>

## 決定
<何を選んだか>

## 理由
<なぜそれを選んだか>

## 却下した代替案
<検討したが選ばなかった案とその理由>

## 影響
<バンドルサイズ・互換性・運用への影響>
```

`add-dependency-safely` スキルは、依存を追加したときにこの形式で ADR を作成する。
